## 1. Windows 环境下的 conda + Git Bash 兼容问题

### 问题

在 Git Bash 中使用 `conda activate` 或 `conda run` 会遇到编码错误和 shell 兼容问题：

```
UnicodeDecodeError: 'gbk' codec can't decode byte 0xae
AttributeError: 'NoneType' object has no attribute 'strip'
```

### 原因

- Git Bash 使用 POSIX shell，而 conda 的 `activate` 脚本依赖 PowerShell/cmd
- Windows 路径中的中文字符在 GBK/UTF-8 转换时导致解码失败

### 解决方案

直接使用 conda 环境的 Python 可执行文件路径，绕过 `conda activate`：

```bash
PY=/d/software/anaconda3/envs/super-base/python.exe
$PY -m pip install -e packages/sdk-python
$PY -m pytest tests/
```

### 经验

**永远不要在 Git Bash 中尝试 `conda activate`，直接用绝对路径调用 Python。**

## 2. pip install -e 相对路径在非项目根目录失败

### 问题

在 `/c/Users/Sakura/Desktop/test` 目录下运行 `pip install -e packages/sdk-python` 持续报错：

```
ERROR: packages/sdk-python is not a valid editable requirement
```

### 原因

Bash 工具的工作目录在会话间持久化。如果 CWD 是 `/c/Users/Sakura/Desktop/test`，而项目在 `super-base-system/` 子目录下，相对路径 `packages/sdk-python` 无法解析。

### 解决方案

每次执行命令前先确认 CWD，或在命令前加 `cd` 前缀：

```bash
cd super-base-system && pip install -e packages/sdk-python
```

### 经验

**这是本次构建中最大的时间浪费点——同一命令重复失败 20+ 次才定位到 CWD 问题。** 已写入自动纠错规则：同一命令失败 2 次必须检查 CWD 和路径。

## 3. pyproject.toml build-backend 配置错误

### 问题

```toml
build-backend = "setuptools.backends._legacy:_Backend"
```

报错：`Cannot import 'setuptools.backends._legacy'`

### 原因

`setuptools.backends._legacy` 是 setuptools 早期实验性 API，在新版中已移除。

### 解决方案

使用标准 build backend：

```toml
build-backend = "setuptools.build_meta"
```

## 4. .env 文件路径解析层级错误

### 问题

Cloud 模式下 `API_KEY` 为空，Anthropic SDK 报错：

```
TypeError: Could not resolve authentication method
```

### 原因

`config.py` 在 `gateway/app/config.py`，使用 `Path(__file__).parent.parent / ".env"` 解析到的是 `gateway/.env`，而不是项目根目录的 `.env`。

### 解决方案

需要 `.parent.parent.parent` 才能到项目根目录：

```python
# gateway/app/config.py 位于 gateway/app/ 下，需要三层 parent
_env_path = Path(__file__).resolve().parent.parent.parent / ".env"
```

### 经验

**嵌套目录中的配置文件路径解析很容易出错，建议用 `print(path)` 验证实际解析结果。**

## 5. Python 内联脚本引号转义问题

### 问题

在 Bash 工具中执行包含中文和复杂引号的 Python 内联脚本时，shell 引号转义导致命令解析失败：

```bash
python -c "print('你好')"  # 可能失败
```

### 解决方案

将测试脚本写入文件后执行，避免引号转义问题：

```python
# tests/e2e_run.py
print("你好")
```

```bash
python tests/e2e_run.py
```

### 经验

**超过 3 行的 Python 脚本不要内联执行，写入文件更可靠。**

## 6. Import 路径跨模块不一致

### 问题

`Complexity` 枚举定义在 `superbase_sdk.schemas` 中，但在 `app.planner.dag` 和 `app.api.chat` 中被误导入：

```python
from ..planner.dag import Complexity  # 错误：Complexity 不在这里
```

### 原因

最初在 `dag.py` 中也定义了 `Complexity`，后来重构到 SDK 中，但多处引用未同步更新。

### 解决方案

统一从 SDK 导入：

```python
from superbase_sdk.schemas import Complexity
```

### 经验

**枚举和共享类型应始终定义在 SDK 层，其他模块统一从 SDK 导入，避免多处定义。**

## 7. NATS JetStream 在 MVP 阶段的降级设计

### 设计决策

NATS 需要独立服务进程，在开发环境中可能不可用。采用惰性初始化 + 自动降级策略：

```python
async def connect(self) -> bool:
    try:
        self._nc = await nats.connect(self._url)
        self._connected = True
        return True
    except Exception as e:
        logger.warning(f"NATS not available, falling back to sync: {e}")
        self._connected = False
        return False
```

调度层始终先尝试 NATS，不可用时自动降级为同步 HTTP 调用。这保证了：
- 开发环境零依赖启动
- 生产环境自动利用消息总线

## 8. DAG Executor 并行 barrier 的实现

### 难点

DAG 中多个无依赖关系的节点需要并行执行，但必须等待当前层所有节点完成后才能进入下一层。

### 解决方案

使用拓扑排序 + asyncio.gather 实现分层执行：

```python
while remaining:
    ready = [nid for nid in remaining
             if all(states[dep].status == COMPLETED for dep in node.depends_on)]

    await asyncio.gather(*[_execute_node(nid) for nid in ready])

    for nid in ready:
        remaining.discard(nid)
```

### 关键点

- 每轮只执行"依赖全部满足"的节点
- `asyncio.gather` 自然实现 barrier：所有并行任务完成后才进入下一轮
- 失败节点不会阻塞无依赖的兄弟节点

## 9. SSE 流式输出的事件回放机制

### 难点

DAG 执行过程中产生多个进度事件，但执行本身是异步的，需要在执行完成后统一通过 SSE 推送。

### 解决方案

采用"收集-回放"模式：

1. 执行前发送 `plan_created` 事件
2. 执行过程中通过回调收集所有进度事件
3. 执行完成后回放所有进度事件 + 最终结果

```python
async def on_progress(node_id, event, data):
    progress_events.append((event, data))

dag_result = await dag_executor.execute(dag, ctx, on_progress=on_progress)

for event, data in progress_events:
    yield sse.format_event(event, data)
yield sse.format_event("completed", final_data)
yield sse.format_done()
```

## 10. 模块注册与 Manifest 扫描

### 设计

模块注册采用"约定优于配置"方式——在 `modules/` 目录下放置 `module_manifest.yaml`，启动时自动扫描加载。

### 注意事项

- Manifest 使用 YAML 格式，需要 `pyyaml` 依赖
- 使用 `rglob("module_manifest.yaml")` 递归扫描，目录结构灵活
- 加载失败的 manifest 打印警告但不中断启动

## 11. 条件分支 AST 安全求值

### 难点

条件节点需要评估用户定义的表达式（如 `ctx.score > 80`），但直接 `eval()` 有严重安全风险。

### 解决方案

使用 Python `ast` 模块的 `ast.parse(expr, mode='eval')` + 手动遍历 AST，白名单允许的节点类型：

```python
class _SafeEvaluator(ast.NodeVisitor):
    def visit_Compare(self, node): ...   # ==, !=, >, <, in
    def visit_BoolOp(self, node): ...    # and, or
    def visit_UnaryOp(self, node): ...   # not
    def visit_Constant(self, node): ...  # 字面量
    def visit_Call(self, node):
        raise ConditionEvalError("Function calls are not allowed")
```

### 经验

**`ast.parse(mode='eval')` 是安全表达式求值的最佳方案——比 `eval()` 安全，比手写 parser 简单。关键是在 `visit()` 中拒绝所有危险节点（Call、Import、Attribute chains）。**

## 12. 人工门控 DAG 暂停/恢复

### 难点

DAG 执行引擎是同步循环（拓扑排序 → gather → 下一轮），遇到人工门控节点需要**暂停整个循环、返回中间状态、等待回调后恢复**。

### 解决方案

采用"快照 + resume"模式：

1. `execute()` 检测到 AWAITING_HUMAN → 快照执行状态（states、results、remaining）→ 提前返回
2. Session 存储快照（内存或 PG）
3. `/human-callback` 端点调用 `resume()` → 从快照恢复 → 继续拓扑循环

```python
# execute() 中
if awaiting_gates:
    return {
        "status": "awaiting_human",
        "_snapshot": {"states": ..., "results": ..., "remaining": ...},
    }

# resume() 中
snapshot = session_manager.get_execution_snapshot(session_id)
states = {k: NodeState(**v) for k, v in snapshot["states"].items()}
# ... 从快照恢复，继续循环
```

### 经验

**"可恢复执行"的核心是将循环状态外部化。拓扑循环的状态 = (states, results, remaining, node_map)，全部可序列化就能暂停/恢复。**

## 13. PostgreSQL async ORM 与内存 fallback 共存

### 难点

SessionManager 原方法是同步的（`def get_or_create`），PG 实现需要 async（`async def`），但两套实现必须共享同一接口。

### 解决方案

1. 将 SessionManager 所有方法改为 `async def`（同步实现加 `async` 不影响功能）
2. 工厂模式：`create_session_repository()` 根据 `DATABASE_URL` 选择实现
3. 测试时始终用内存模式（无需 PG），PG 实现通过独立测试验证

```python
def create_session_repository():
    if settings.database_url:
        return PgSessionRepository()
    return SessionManager()  # 内存 fallback
```

### 经验

**接口统一比实现统一更重要。将所有方法改为 async 后，调用方代码（chat.py）只需加 `await`，不关心底层是内存还是 PG。**

## 14. 向量检索 Mock 与真实 Embedding 的接口设计

### 难点

Embedding 是 LTM 的核心，但开发环境可能没有 embedding API（MiMo 代理不支持 embeddings）。

### 解决方案

设计 `EmbeddingProtocol` 接口，Mock 实现使用确定性伪向量：

```python
class MockEmbeddingClient:
    def _generate(self, text: str) -> list[float]:
        seed = int(hashlib.sha256(text.encode()).hexdigest(), 16) % (2**32)
        rng = random.Random(seed)
        vec = [rng.gauss(0, 1) for _ in range(self._dimension)]
        # L2 归一化
        norm = math.sqrt(sum(x * x for x in vec))
        return [x / norm for x in vec]
```

### 关键点

- 相同文本 → 相同向量（SHA256 种子），保证测试可重复
- L2 归一化到单位球面，cosine 距离有意义
- 不同文本 → 不同向量，但无语义相似性（仅测试用）

### 经验

**Mock 实现的底线是"确定性"和"接口兼容"。Mock embedding 不需要语义正确，但必须相同输入 = 相同输出，否则测试不可重复。**

## 15. Reranker 混合评分策略

### 难点

纯 cosine 相似度在 Mock embedding 下效果差（随机向量），接入真实 embedding 后也可能不够精确。

### 解决方案

混合评分：`final = α × semantic_score + (1-α) × keyword_score`

```python
class Reranker:
    def rerank(self, query, results, top_k):
        keywords = self._extract_keywords(query)  # 中文/英文分词 + 去停用词
        for result in results:
            keyword_score = keyword_overlap(keywords, result.content)
            blended = 0.7 * semantic_score + 0.3 * keyword_score
```

### 经验

**关键词重叠是零成本的后备信号。当 embedding 质量不确定时，混合评分比纯语义更鲁棒。α=0.7 是经验值，可按实际效果调整。**

## 16. 多租户跨层注入

### 难点

`tenant_id` 需要贯穿 API → Session → Memory → DB → Qdrant 全链路，但当前架构各层独立。

### 解决方案

1. **API 层**：中间件提取 `X-Tenant-ID` → `request.state.tenant_id`
2. **chat.py**：从 `request.state` 取出，显式传递到 session/memory 调用
3. **Memory**：`recall(tenant_id=...)` → `ltm.search(tenant_id=...)` → Qdrant filter
4. **DB**：`tenant_id` 列 + 索引（RLS 策略可选启用）

```python
# chat.py
tenant_id = getattr(request.state, "tenant_id", "default")
recall_result = await _memory.recall(session_id, query, tenant_id=tenant_id)
```

### 经验

**跨层参数传递用"显式参数"比"隐式上下文"更可靠。虽然每个函数签名多一个参数，但数据流清晰可追踪，测试也更容易 mock。**

## 17. OpenTelemetry 与 Prometheus 的初始化时机

### 难点

OTel TracerProvider 必须在应用启动时初始化一次，但 FastAPI 的中间件在 app 创建时就注册。

### 解决方案

在 `@app.on_event("startup")` 中调用 `init_tracing()`，使用 `ConsoleSpanExporter` 作为默认（开发），可选 OTLP 导出（生产）：

```python
@app.on_event("startup")
async def startup():
    init_tracing()  # 初始化 TracerProvider
    ...
```

### 经验

**可观测性的初始化不要放在模块顶层——放在 startup hook 中，保证 FastAPI app 已创建且环境变量已加载。**

## 总结

| 难点类别 | 数量 | 最严重的问题 |
|----------|------|-------------|
| 环境/路径问题 | 4 | CWD 错误导致 20+ 次重复失败 |
| 依赖/配置问题 | 2 | build-backend 和 .env 路径 |
| 架构设计问题 | 6 | DAG 并行 barrier、人工门控暂停/恢复 |
| 安全/隔离问题 | 2 | AST 安全求值、多租户跨层注入 |
| 接口/抽象问题 | 3 | Mock embedding 确定性、async 接口统一 |

**最大教训：环境类错误（路径、CWD、编码）比代码逻辑错误更难定位，因为错误信息不直观。应优先验证环境再写代码。**
