## 1. 熔断器（Circuit Breaker）

### 问题

模块不可用时，每次请求仍会尝试调用，浪费超时等待时间（默认 60s），拖慢整个 DAG 执行。

### 机制

三态熔断器，按模块独立管理：

```
CLOSED（正常）──失败 N 次──→ OPEN（熔断，拒绝调用）
                                │
                          等待 recovery_timeout
                                │
                                ▼
                           HALF_OPEN（试探）
                           ├── 试探成功 → CLOSED
                           └── 试探失败 → OPEN
```

### 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `failure_threshold` | 5 | 连续失败 N 次触发熔断 |
| `recovery_timeout` | 30s | 熔断后等待时间 |
| `half_open_max_calls` | 1 | 半开状态允许的试探调用数 |

### API

```
GET  /api/v1/circuit-breaker              # 查看所有模块熔断状态
POST /api/v1/circuit-breaker/{module_id}/reset  # 手动重置熔断器
```

### 代码位置

`gateway/app/resilience/circuit_breaker.py`

## 2. 死信队列（Dead Letter Queue）

### 问题

重试耗尽的任务直接丢失，无法追踪失败原因，无法手动重放。

### 机制

失败任务自动进入 DLQ，支持按模块过滤、手动重放：

```
模块执行失败（重试耗尽）
    → 自动生成 DeadLetter 记录
    → 存入内存队列（生产环境应持久化到 PostgreSQL）
    → 运维人员通过 API 查看、排查、重放
```

### DeadLetter 数据结构

```json
{
  "dlq_id": "dlq_abc123",
  "task_id": "node_1",
  "module_id": "echo",
  "error": "模块 echo 执行超时（60s）",
  "retry_count": 2,
  "payload": {"query": "hello", "session_id": "sess_xyz"},
  "created_at": "2026-05-19T10:00:00Z",
  "replayed": false
}
```

### API

```
GET  /api/v1/dlq                          # 查看死信队列
GET  /api/v1/dlq?module_id=echo          # 按模块过滤
POST /api/v1/dlq/{dlq_id}/replay         # 重放死信任务
```

### 代码位置

`gateway/app/resilience/dead_letter.py`

## 3. 全局降级（Graceful Degradation）

### 问题

所有业务模块均不可用时，用户请求无任何响应。

### 机制

DAG 执行完成后，检测所有节点状态：
- 如果所有模块节点均 failed/skipped → 触发降级
- 返回友好提示 + 记录到 DLQ
- 后续可扩展为调用基座模型直接生成回答（跳过模块调度）

```
用户请求 → 意图识别 → DAG 执行
                         │
                    所有模块失败？
                         │
                    ┌────┴────┐
                    │ 是      │ 否
                    ▼         ▼
              降级响应    正常聚合结果
        "所有模块不可用"
        （记录 DLQ）
```

### 代码位置

`gateway/app/api/chat.py` → `_execute_and_respond()`

## 4. 模块健康检查（Health Check）

### 问题

模块进程崩溃或端口未监听时，调度层无法提前感知，导致无效调用。

### 机制

Dispatcher 提供 `health_check()` 方法，通过 `GET /health` 端点探测模块存活状态：

```
GET /api/v1/modules/health
→ {
    "modules": {
      "echo": {"healthy": true, "circuit": "closed", "port": 8001},
      "weather": {"healthy": false, "circuit": "open", "port": 8002}
    }
  }
```

### 代码位置

`gateway/app/dispatcher/http_dispatcher.py` → `health_check()`

## 5. 可配置超时（Configurable Timeout）

### 问题

不同模块执行时间差异大（echo 毫秒级 vs 代码审查秒级），统一超时不合理。

### 机制

每个模块在 `module_manifest.yaml` 中声明 `timeout_seconds`，Dispatcher 按模块独立设置 HTTP 超时：

```yaml
# echo 模块：快速
runtime:
  timeout_seconds: 30

# code_review 模块：较慢
runtime:
  timeout_seconds: 120
```

### 代码位置

`gateway/app/dispatcher/http_dispatcher.py` → `dispatch()` 的 `timeout` 参数

## 6. 指数退避重试（Exponential Backoff Retry）

### 问题

模块临时故障（如网络抖动）时，立即重试可能仍然失败。

### 机制

DAG Executor 节点执行失败后，按指数退避策略重试：

```
第 1 次失败 → 等待 1s → 重试
第 2 次失败 → 等待 2s → 重试
第 3 次失败 → 进入 DLQ，标记 FAILED
```

| 参数 | 值 |
|------|-----|
| 最大重试次数 | 2（可配置） |
| 退避策略 | `min(2^attempt, 4)` 秒 |
| 总最大耗时 | ~7s（1+2+4 超时） |

### 代码位置

`gateway/app/executor/dag_executor.py` → `_execute_node()` 重试循环

## 整体容错流程

```
用户请求
    │
    ▼
意图识别 → DAG 编排
    │
    ▼
DAG Executor 逐节点执行
    │
    ├── 熔断器检查 → OPEN？ → 跳过，直接进 DLQ
    │
    ├── 模块调用
    │     ├── 成功 → 重置熔断器，继续下一节点
    │     └── 失败 → 指数退避重试（最多 2 次）
    │               └── 仍然失败 → 记录熔断器 + 进入 DLQ
    │
    ▼
结果聚合
    │
    ├── 有成功节点 → 正常返回结果
    └── 全部失败 → 全局降级响应 + DLQ 记录
```

## 7. 条件分支执行（Condition Branch）

### 问题

复杂业务流程需要根据运行时数据选择不同执行路径（如评分 > 80 走快速通道，否则走人工审核）。

### 机制

基于 Python `ast` 模块的安全表达式求值器 + DAG 分支跳过：

```
条件节点评估表达式（如 "ctx.score > 80"）
    ├── True  → 执行 true_branch，跳过 false_branch（SKIPPED）
    └── False → 执行 false_branch，跳过 true_branch（SKIPPED）

下游节点依赖被跳过的节点 → 自动 SKIPPED（依赖不满足）
```

支持运算符：`==` `!=` `>` `<` `>=` `<=` `in` `contains` `and` `or` `not`

### 代码位置

- 求值器：`gateway/app/planner/condition_evaluator.py`
- 分支跳过：`gateway/app/executor/dag_executor.py` → `_skip_branches()`

## 8. 人工介入节点（Human Gate）

### 问题

某些任务需要人工审批（如敏感操作、高风险决策），不能完全自动化。

### 机制

DAG 中插入 HUMAN_GATE 节点，执行到该节点时暂停，等待人工回调后恢复：

```
DAG 执行 → HUMAN_GATE 节点
    → 状态设为 AWAITING_HUMAN
    → 快照执行状态（states/results/remaining）
    → 提前返回 status="awaiting_human"
    → 独立分支继续执行（不阻塞）
    
人工回调 POST /sessions/{id}/human-callback
    → action: approve（继续）/ reject（跳过下游）/ modify（带修改数据继续）
    → 从快照恢复 → 继续拓扑循环
```

超时策略：可配置 `timeout_seconds` + `default_action`（默认 reject）。

### API

```
POST /api/v1/sessions/{session_id}/human-callback
Body: {"action": "approve|reject|modify", "data": {...}}
```

### 代码位置

`gateway/app/executor/dag_executor.py` → `_start_human_gate()` + `resume()`

## 9. PostgreSQL 持久化

### 问题

所有运行时状态（会话、消息、死信）均为内存存储，重启即丢失。

### 机制

Repository 模式 + 工厂 + 内存 fallback：

```
DATABASE_URL 设置？──→ PgSessionRepository / PgDeadLetterQueue
                     │
未设置 ──────────→ SessionManager（内存） / DeadLetterQueue（内存）
                     + 打印 WARNING
```

表结构：`sessions`（session_id, tenant_id, created_at, pending_gate_info, execution_snapshot）+ `messages`（FK→sessions）+ `dead_letters`

### 代码位置

- ORM：`gateway/app/db/models.py`
- Engine：`gateway/app/db/engine.py`
- Factory：`gateway/app/db/factory.py`
- PG Session：`gateway/app/session/pg_repository.py`
- PG DLQ：`gateway/app/resilience/pg_dlq.py`

## 10. 短期记忆（STM）

### 问题

会话历史只存不读，模块执行时看不到对话上下文。

### 机制

滑动窗口 + token 预算 + Redis/内存双实现：

```
STM.recall(session_id, query)
    → 取最近 N 轮对话（默认 10 轮）
    → 按 token 预算截断（默认 2000 tokens）
    → 注入到 base_context.conversation_history
```

Redis 实现：`LPUSH` + `LTRIM` 保留最近 N 条，TTL 24h 自动过期。

### 代码位置

- 接口 + 内存：`gateway/app/memory/stm.py`
- Redis：`gateway/app/memory/redis_stm.py`

## 11. 长期记忆（LTM）

### 问题

有价值的对话内容跨会话后丢失，无法跨会话召回。

### 机制

向量存储 + 语义检索 + 智能晋升：

```
store_interaction(session_id, user_msg, assistant_msg)
    → 始终存入 STM
    → 长度 > 100 / 含关键词（重要/决策/remember）/ 回复 > 150 字符 → 存入 LTM

recall(session_id, query)
    → STM.recall() → 会话历史
    → LTM.search() → 语义检索
    → Deduplicate → Rerank（语义 70% + 关键词 30%）→ IntentFilter
    → 合并返回 {conversation_history, retrieved_memories}
```

### 代码位置

- LTM 接口 + 实现：`gateway/app/memory/ltm.py`
- Reranker + IntentFilter：`gateway/app/memory/rag.py`
- 统一管理：`gateway/app/memory/manager.py`

## 12. 可观测性

### 问题

零可观测性：无分布式追踪、无指标、无结构化日志、无请求中间件。

### 机制

三层可观测（不依赖外部基础设施）：

| 层 | 技术 | 输出 |
|---|---|---|
| 追踪 | OpenTelemetry | Span 链（chat → intent → memory → dag → node） |
| 指标 | prometheus_client | `/metrics` 端点，10 个指标（Counter + Histogram） |
| 日志 | Python logging + JSON | 结构化日志（request_id, method, path, status, duration_ms） |

关键指标：`http_requests_total`、`dag_execute_duration_seconds`、`module_dispatch_duration_seconds`、`memory_recall_duration_seconds` 等。

### 代码位置

- 追踪：`gateway/app/observability/tracing.py`
- 指标：`gateway/app/observability/metrics.py`
- 中间件：`gateway/app/main.py` → `observability_middleware`

## 13. 多租户隔离

### 问题

所有数据全局共享，无租户隔离，多团队无法共用基础设施。

### 机制

四层隔离：

| 层 | 方案 |
|---|---|
| API | `X-Tenant-ID` 请求头 → `request.state.tenant_id` |
| 限流 | 滑动窗口 RateLimiter（默认 60 req/60s per tenant） |
| PostgreSQL | `tenant_id` 列 + 索引（RLS 策略可选） |
| LTM/Qdrant | 搜索时 filter `tenant_id` |

超限返回 429，响应头包含 `X-RateLimit-Remaining`。

### 代码位置

- 中间件：`gateway/app/tenant/middleware.py`

## 完整容错与优化流程

```
用户请求（X-Tenant-ID）
    │
    ▼
TenantMiddleware → RateLimiter 检查
    │ 429？→ 拒绝
    ▼
意图识别（OTel span: intent.classify）→ DAG 编排
    │
    ▼
Memory.recall（STM + LLM 语义检索 + Reranker + IntentFilter）
    │ 注入 conversation_history + retrieved_memories
    ▼
DAG Executor 逐节点执行（OTel span: dag.execute）
    │
    ├── 条件节点 → AST 求值 → 分支跳过
    ├── 人工门控 → AWAITING_HUMAN → 快照 → 等待回调
    ├── 熔断器检查 → OPEN？ → 跳过，进 DLQ
    ├── 模块调用（OTel span: module.dispatch + Prometheus histogram）
    │     ├── 成功 → 重置熔断器
    │     └── 失败 → 指数退避重试 → DLQ
    ▼
结果聚合 + Memory.store_interaction（智能晋升 LTM）
    │
    ├── 有成功节点 → 正常返回
    └── 全部失败 → 全局降级响应 + DLQ 记录
```
